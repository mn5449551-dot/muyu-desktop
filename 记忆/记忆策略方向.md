# 记忆功能策略 v2.1（最终完整版）

> 适用产品：桌面宠物 · 情感陪伴 + 解压应用
> 版本说明：在 v2.0 基础上，补充主动发话机制、记忆管理界面、多宠物数据隔离策略、向量检索替代方案

---

## 一、整体架构

### 核心定位

让宠物"像一个真正认识用户的朋友"——记得你说过的话，记得你的行为习惯，在合适的时机主动提起，产生真实的情感连接。

### 完整工作流

```
用户输入（文字 / 敲击行为）
          ↓
    记忆召回（四层）
          ↓
    分级注入 · Prompt 拼接
          ↓
        模型回复
          ↓
    异步记忆更新
    ├── 中期摘要（每12轮）
    ├── 长期档案（绑定摘要触发）
    └── 行为记忆（实时写入 + 每周分析）
```

### 四层记忆模型总览

| 记忆类型 | 定位 | 生命周期 | 存储位置 |
|---------|------|---------|---------|
| 短期记忆 | 当前对话的"即时注意力" | 本次对话轮 | 内存 / 上下文窗口 |
| 中期记忆 | 本次会话的"整体脉络" | 会话期间（全量保留） | 会话表 |
| 长期记忆 | AI"真正认识这个人" | 永久（状态类带衰减） | 用户档案表 + 事件表 |
| 行为记忆 | 用户敲击习惯与情绪规律 | 永久 | 行为日志表 + 规律表 |

**一句话总结**

- 短期 = 刚说的话
- 中期 = 今天聊了啥
- 长期 = 你是谁、你经历过什么
- 行为 = 你怎么用我、什么时候情绪最强

---

## 二、短期记忆

### 定位

当前对话的"即时注意力"，保证接话自然、不脱节。

### 实现策略

| 参数 | 值 | 说明 |
|------|-----|------|
| 保留轮数 | 默认 12 轮（可配置，范围 8-15） | 超出后从最早开始丢弃 |
| 处理方式 | 原样保留 | 不做任何压缩，直接塞入上下文 |
| 计数方式 | 仅计有实质内容的轮次 | `is_meaningful = 0` 的消息不计入 |

### 边界条件

- 不足 12 轮时全部保留，不触发摘要
- 水消息（"哈哈"、纯敲击）标记 `is_meaningful = 0`，不计入有效轮数，但仍存储用于行为分析

---

## 三、中期记忆

### 定位

本次会话的"整体脉络"，保证一整段聊天都连贯，不丢失主题。

### 触发策略

| 条件 | 说明 |
|------|------|
| 轮数触发 | 每累计 12 轮有效对话 |
| 时间兜底 | 距上次摘要超过 7 天，强制生成 |
| 话题检测延迟 | 话题未结束时，最多延迟 5 轮再触发 |
| 最小有效内容 | 低于 5 轮有效对话，强制触发时摘要可为空或极简 |

### 话题检测逻辑

```python
def should_trigger_summary(session_id):
    recent_5 = get_recent_messages(session_id, limit=5)
    is_same_topic = llm_check_topic_continuity(recent_5)  # 轻量分类

    if is_same_topic and session.delay_count < 5:
        session.delay_count += 1
        return False
    else:
        session.delay_count = 0
        return True
```

### 摘要内容（5 个维度）

```
1. 心情状态        → 用户当前情绪（正面 / 负面 / 平淡）
2. 今天发生的事    → 具体事件，尽量保留关键细节
3. 重要话题        → 反复提到的主题
4. 未解决的情绪/诉求 → 用户提出但未得到充分响应的内容
5. 对宠物的态度    → 今天的互动模式（调侃 / 依赖 / 倾诉 / 闲聊）
```

> 第 5 维度直接影响宠物下次打招呼的语气，是桌面宠物场景的特有价值。

### 摘要后的上下文结构

```
[角色设定]
[长期记忆：用户档案摘要]
[中期记忆：本次会话摘要]
[短期对话：最近 12 轮]
```

## 四、长期记忆

### 定位

AI"真正认识这个人"。存储用户的基本信息、偏好、习惯、关系和重要事件。

### 更新时机

绑定摘要触发，不单独运行——每次生成中期摘要时，同步检查并更新长期档案，避免每轮调用 LLM 抽取信息。

### 时效性分类

| 字段类型 | 示例字段 | 时效策略 |
|---------|---------|---------|
| 永久有效 | 基本信息、偏好、关系 | 不衰减 |
| 状态类（带衰减） | current_mood、health_status、current_status、recent_goal | 按字段设定过期天数 |
| 事件类 | 关键事件 | 按 importance 打分，低分事件 90 天后降权 |

**状态类字段过期天数**

| 字段 | 过期天数 | 原因 |
|------|---------|------|
| current_mood | 7 天 | 情绪变化快 |
| health_status | 14 天 | 病情通常短期恢复 |
| current_status | 30 天 | 项目 / 工作状态变化周期 |
| recent_goal | 60 天 | 目标周期较长 |
| recent_event | 60 天 | 事件影响消退 |

过期字段标记 `is_active = 0`，注入前过滤；数据库保留不删（用于事件回溯和冲突检测）。

## 五、行为记忆

### 定位

记录用户敲击行为的规律，挖掘行为与情绪的关联，让宠物能在合适时机说出有温度的话。

### 当前阶段：仅支持敲击动作

目前只有 `tap` 一种 action_type，后续新增动作时直接扩展字段，不影响现有逻辑。

### 有效信号提取

| 信号 | 提取方式 | 推断含义 |
|------|---------|---------|
| 单次持续时长 | action_duration | 越长 = 越需要解压 |
| 敲击频率（次/秒） | action_count ÷ duration | 越快 = 情绪越强烈 |
| 当日累计次数 | 日内汇总 | 超出个人均值 = 今天压力偏大 |
| 发生时段 | hour_of_day | 深夜高频 = 可能焦虑/失眠 |
| 敲完后是否说话 | 关联 session 消息 | 有倾诉行为 = 情绪有出口 |

> 无法精确区分"焦虑"和"愤怒"，行为信号只能判断情绪强度，具体类型需结合后续对话内容补充。

### 行为规律分析（每周异步运行）

```python
def analyze_behavior_pattern(pet_id):
    data = get_behavior_log(pet_id, days=30)

    # 1. 高频敲击时段
    peak_hours = find_peak_hours(data)           # e.g. [22, 23, 0]

    # 2. 个人化异常阈值
    personal_avg = calculate_daily_avg(data)
    abnormal_threshold = personal_avg * 1.5

    # 3. 敲击后情绪关联
    post_action_pattern = correlate_action_and_chat(data)
    # e.g. "高频敲击后，60% 概率接着聊工作压力"

    update_behavior_pattern(pet_id, peak_hours, abnormal_threshold, post_action_pattern)
```

### 行为记忆注入示例

```
[行为记忆]
- 用户常在 22:00-24:00 高频敲击（推测夜间压力较大）
- 本次敲击持续 97 秒，频率超过个人均值 1.8 倍
- 历史规律：密集敲击后，60% 概率接着倾诉工作压力
```

## 六、主动发话机制

### 触发类型总览

| 触发类型 | 触发条件 | 典型话术方向 | 优先级 |
|---------|---------|------------|-------|
| 时间触发 | 生日、纪念日、提醒事项 | "今天是你的生日诶" | 最高 |
| 情绪触发 | 上次对话有未解决情绪 | "昨天那件事，后来怎么样了？" | 高 |
| 行为触发 | 敲击异常停止后 | "敲了这么久，还好吗？" | 中 |
| 缺席触发 | 超过 N 天没打开 | "你去哪了，我想你了" | 低 |

> 同时满足多个条件时，按优先级取最高的一个触发，不叠加。

---

### 6.1 行为触发（敲击停止后）

**检测逻辑**

```
1. 记录最后一次敲击时间 last_tap_ts
2. 每隔 3 秒检测一次
3. 当前时间 - last_tap_ts > 静默阈值 → 进入触发判断

静默阈值（按敲击时长分级）：
- 敲击时长 < 30 秒     → 不触发（随手敲，不打扰）
- 敲击时长 30s - 2min  → 停止后等 5 秒触发
- 敲击时长 > 2min      → 停止后等 3 秒触发

触发条件（需同时满足）：
- 本次敲击超过个人均值 × 1.5（异常才说话，正常不打扰）
- 距上次主动发话 > 30 分钟
- 用户当前没有正在输入
```

**话术策略（结合长期记忆）**

```
白天工作时段（9:00-18:00）       → "刚才发生什么了？"
深夜（22:00 以后）               → "这么晚了还没睡，睡不着吗？"
档案含 work_routine = "经常加班" → "又在加班？"
上次摘要有未解决情绪             → 接着上次话题问
```

---

### 6.2 时间触发

**触发节点**

| 节点 | 触发条件 | 说明 |
|------|---------|------|
| 用户生日 | birthday 字段匹配当天日期 | 当天 app 启动时触发，过了当天不再提 |
| 第一次聊天纪念日 | key_events 最早记录的周年 | 每年自动标记 |
| 用户自设提醒 | reminder 字段到期 | 到时间触发，不依赖系统通知 |
| 惯常打开时段 | 行为规律中的 peak_hours | 用户在习惯时段打开，宠物主动打招呼 |

**边界条件**

- 生日/纪念日：**只在当天有效**，次日打开不再提（避免"昨天是你生日"的尴尬）
- 提醒事项：触发一次后标记 `is_triggered = 1`，不重复触发
- 所有时间触发：均在 app 启动时检测，不做系统级推送通知

---

### 6.3 情绪触发

```
触发条件（需同时满足）：
- 上次会话摘要中"未解决的情绪/诉求"字段非空
- 距上次对话 > 12 小时（话题已有时间沉淀）
- 距上次对话 < 7 天（话题未过期）

触发时机：用户打开 app 时

示例：
上次摘要：用户在和朋友闹矛盾，情绪未消化
→ 宠物主动问："昨天你说的那件事，后来有和好吗？"
```

---

### 6.4 缺席触发

```
触发条件：距上次打开 app 超过 3 天
触发时机：用户重新打开 app 的那一刻

话术分级：
- 3-7 天   → "你去哪了，我想你了"（轻度想念）
- 7-14 天  → "好久没见你了，还好吗？"（适度担心）
- > 14 天  → 不主动提缺席，正常打招呼（追太久显粘）
```

---

### 6.5 防打扰机制（全局）

```
全局规则：
- 同一次 app 使用周期内，主动发话最多 1 次
- 用户正在输入时，不触发任何主动话术
- 距上次用户主动发消息 < 5 分钟，不触发（对话还热着）
- 行为触发的冷却时间：同一会话内 30 分钟
```

## 七、分级注入策略

### 三级注入机制

| 注入级别 | 内容 | 注入时机 | 预估 Token |
|---------|------|---------|-----------|
| 强注入 | 用户称呼、角色关系、核心偏好 | 每次对话都带 | ~100 |
| 弱注入 | 近期状态、情绪（未过期） | 状态未过期时带入 | ~150 |
| 触发注入 | 历史事件、行为规律 | 语义相关或行为异常时触发 | ~200 |

### 历史事件检索（关键词匹配替代向量检索）

```python
def trigger_inject_events(user_message, pet_id):
    keywords = extract_keywords(user_message)  # 轻量分词，不需要 LLM

    events = []
    for kw in keywords:
        results = db.query("""
            SELECT * FROM key_events
            WHERE pet_id = ?
              AND is_active = 1
              AND (event_content LIKE ? OR emotion_tag LIKE ?)
            ORDER BY importance DESC, ts DESC
            LIMIT 2
        """, [pet_id, f"%{kw}%", f"%{kw}%"])
        events.extend(results)

    return deduplicate(events)[:2]
```

> 后期数据量变大时，可引入 sqlite-vec 做向量检索，接口层不变，替换实现即可。

### 注入判断标准

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 个人偏好 | "我喜欢吃火锅" | ✅ 写入档案 |
| 个人状态 | "我今天心情不好" | ✅ 写入状态（7天衰减） |
| 个人事实 | "我是程序员" | ✅ 写入档案 |
| 关系/互动 | "叫我大哥" | ✅ 强注入字段 |
| 闲聊寒暄 | "今天天气不错" | ❌ 不写入 |
| 纯敲击互动 | 敲击动作 | ❌ 不写入对话记忆，写入行为日志 |
| 无意义 | "哈哈哈" | ❌ is_meaningful = 0 |

## 八、记忆管理界面

### 用户可查看的内容

```
记忆管理页面展示：

1. 基础档案
   - 宠物"认识"的你：昵称、职业、偏好、习惯等
   - 每条记录显示：来源（你说的 / 宠物推断）、更新时间

2. 宠物互动设定（按 pet_id 展示）
   - 你叫它什么、它叫你什么、你们的角色关系

3. 它记住的事（关键事件列表）
   - 按时间倒序
   - 显示事件内容、情绪标签、重要性

4. 状态信息
   - 过期的字段灰显，注明"已过期，不再使用"
```

### 用户可执行的操作

| 操作 | 实现方式 | 说明 |
|------|---------|------|
| 删除某条记忆 | 标记 `is_active = 0` | 软删除，不物理删除 |
| 修改档案字段 | 更新 value，source 改为 user_input | 手动修改优先级最高 |
| 清空全部记忆 | 所有字段批量 `is_active = 0` | 提供二次确认 |
| 查看已忘记的内容 | 展示 `is_active = 0` 的字段（灰显） | 透明化，用户知道删了什么 |

### 边界处理

- 用户删除的字段，仍保留在数据库（`is_active = 0`），用于冲突检测防止被重复误判写入
- 用户手动修改的字段，`source = user_input`，后续自动推断不覆盖手动输入
- 过期字段在界面上灰显并标注"已过期"，但不主动删除，用户可手动清理

## 九、言行冲突检测与问询

### 触发逻辑（滑动窗口计数）

```python
def check_conflict(field_key, new_inferred_value, pet_id):
    # 取最近20次涉及该字段的对话推断
    recent = get_recent_inference(field_key, pet_id, limit=20)

    if len(recent) < 5:
        return  # 样本不足，不触发

    conflict_count = sum(1 for r in recent if r.value != profile[field_key])
    conflict_rate = conflict_count / len(recent)

    if conflict_rate >= 0.6:
        trigger_conflict_query(field_key)
```

### 冲突场景

| 冲突类型 | 示例 |
|---------|------|
| 档案 vs 推断 | 档案写"设计师"，但多次对话推断出"程序员" |
| 自述 vs 行为 | 说"在减肥"，但连续多次聊火锅烧烤 |
| 历史 vs 当前 | 之前常提"跑步"，近1个月完全不再提 |

### 问询策略

| 维度 | 设计 |
|------|------|
| 问询方式 | 委婉关心式："我记得你之前说……最近是不是有变化？" |
| 触发阈值 | 滑动窗口内冲突率 ≥ 60%，且样本 ≥ 5 次 |
| 频率限制 | 同一字段每月最多问询 1 次 |
| 用户确认 | 以用户说的为准，字段更新，计数重置 |
| 用户否认 | 保持原档案，计数重置 |
| 用户忽略 | 记录 `ignore_future = 1`，不再主动问询，但仍正常记忆 |

## 十、边界情况汇总

| 场景 | 处理方案 |
|------|---------|
| 会话消息量少，触发不了摘要 | 有效对话 < 12 轮不触发；7天强制触发，摘要可为空或极简 |
| 话题跨轮被截断 | 话题检测 + 最多延迟 5 轮，超过强制触发 |
| 水消息 / 纯敲击 | is_meaningful = 0，不计入有效轮数，写入行为日志 |
| 状态类字段过期被召回 | 注入前检查 is_active，过期字段不注入，保留用于回溯 |
| 多宠物切换 | 基础档案全局共享，互动信息按 pet_id 隔离 |
| 用户要求忘掉某件事 | 软删除（is_active = 0），界面灰显，仍用于冲突检测 |
| 生日当天未打开 app | 次日打开不再提，只在当天有效 |
| 多个主动发话条件同时满足 | 按优先级取最高一个，不叠加 |
| 用户正在输入时行为异常 | 不触发主动话术，等用户停止输入后再判断 |

## 十一、数据库表结构

### 用户档案表（长期记忆）

```sql
CREATE TABLE user_profile (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  pet_id      TEXT NOT NULL DEFAULT 'global',   -- 'global'=共享；具体 pet_id=隔离
  key         TEXT NOT NULL,
  value       TEXT NOT NULL,
  source      TEXT DEFAULT 'user_input',         -- user_input / auto
  is_verified INTEGER DEFAULT 0,
  is_active   INTEGER DEFAULT 1,                 -- 0=过期或用户删除
  field_type  TEXT DEFAULT 'permanent',          -- permanent / status
  expire_days INTEGER DEFAULT NULL,              -- NULL=永不过期
  updated_at  INTEGER NOT NULL,
  UNIQUE(pet_id, key)
);
```

## 十四、验收标准

| 场景 | 预期表现 |
|------|---------|
| 用户说"叫我大哥" | 下次对话自然称呼"大哥" |
| 用户聊了20轮工作压力 | 中期摘要正确记录，下次延续不脱节 |
| 用户两年前说"在备考" | 字段已过期，不注入，不出现奇怪回复 |
| 敲击超过个人均值 1.5 倍 | 停止后 3-5 秒，宠物主动询问 |
| 用户生日当天打开 app | 宠物主动提起生日 |
| 用户3天没打开 | 重新打开时宠物说"你去哪了" |
| 上次聊了未解决的矛盾 | 次日打开，宠物主动跟进 |
| 用户删除一条记忆 | 软删除，界面灰显，不再注入 |
| 用户A的宠物1和宠物2 | 偏好档案共享，称呼关系各自独立 |
| 用户否认档案中的信息 | 字段更新为用户说的，冲突计数重置 |
