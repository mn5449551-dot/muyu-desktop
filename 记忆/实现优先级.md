# 记忆功能实现优先级

> 基于 v2.1 方案评审结论，以当前 v1.0 为起点，分四个阶段渐进实现。
> 每阶段可独立上线、独立回归，不依赖后续阶段。

---

## 当前状态（v1.0 已实现）

- 单层滚动摘要：20条触发，摘要注入 system prompt
- `memory_summaries` 表：无角色隔离（session_id='default'）
- `user_profile` 表：key-value，用户手动填写
- 无主动发话，无记忆管理界面

> ✅ P0 / P1 / P2 全部完成（2026-02-28）

---

## ✅ P0 — 稳定地基（已完成）

**目标：把现有实现做对，堵住上线前的工程风险。**

### ✅ 0-1 Schema 迁移方案
- 写一份 `db-migration.md`：从当前 2 张表 → 目标表结构的 ALTER 语句 + 回滚语句
- 原则：只加字段/表，不删旧字段，保证可回滚
- 迁移时机：app 启动时 `runMigrations()` 按版本号顺序执行

### ✅ 0-2 角色记忆隔离
- `memory_summaries` 加 `character_id` 字段（NOT NULL, DEFAULT 'baihu'）
- 现有数据迁移：UPDATE SET character_id='baihu' WHERE character_id IS NULL
- 摘要触发、查询、注入全部带 character_id 过滤

### ✅ 0-3 时间语义统一
- 所有时间戳存 UTC（`Date.now()` 或 `new Date().toISOString()`）
- 显示层统一用本地时区转换
- "3天缺席"判断基于 UTC 日期差，不依赖本地凌晨

### ✅ 0-4 成本上限保护
- 记一次摘要的 token 消耗（input + output），写入 `summary_history` 表
- 单次摘要 input 超过 8000 tokens 时截断最早消息，不报错不中断
- 每日摘要调用次数上限：10次/角色（防止异常循环触发）

## ✅ P1 — 记忆核心升级（已完成）

**目标：让宠物真正"认识"用户，摘要从非结构化 → 半结构化。**

### ✅ 1-1 结构化用户档案自动提取
- 摘要生成后，额外触发一次轻量 LLM 调用，从对话中提取：
  ```
  姓名、称呼偏好、职业、近期状态、重要偏好（JSON）
  ```
- 提取结果 merge 入 `user_profile`（不覆盖用户手动填写的字段）
- 用 Doubao Lite 小模型（低成本），提取失败静默跳过

### ✅ 1-2 角色关系状态
- `character_memory` 表（每角色一行）：`relationship_stage`, `inside_jokes(JSON)`, `unresolved_threads(JSON)`
- 摘要触发时同步更新（绑定现有触发点，不新增触发逻辑）
- 注入顺序：角色人设 → 用户档案 → 关系状态 → 摘要 → 当前时间

### ✅ 1-3 统一触发判定器
- 单函数 `shouldTriggerSummary(characterId)` 返回 true/false
- 判定条件：未摘要消息 ≥ 20 OR（关闭时 ≥ 10）
- 防重入：Set 标记正在处理的 characterId（已有，确认覆盖所有路径）

---

## ✅ P2 — 体验提升（已完成）

**目标：用户能管理记忆，宠物有基础主动行为。**

### ✅ 2-1 记忆管理界面
- 设置页新增"记忆"Tab：列出所有摘要条目（时间 + 内容预览）
- 支持删除（软删除：加 `deleted_at` 字段，查询过滤）
- 用户删除后的文案："已隐藏，不再影响对话"（明确告知）

### ✅ 2-2 全局静默开关
- 设置项：`proactive_enabled`（布尔），默认关
- 所有主动发话逻辑先检查此开关，关闭时直接跳过

### ✅ 2-3 基础主动发话（仅 2 种触发，不做复杂推断）
- **缺席触发**：距上次打开 > 3天，打开时宠物说一句"你去哪了"
- **生日触发**：用户档案中有生日字段，当天打开触发一次
- 两种触发均有：同天只触发一次的去重锁（存 `app_state` 表）
- **不做**：敲击频率情绪推断、缺席追问（> 14天停止）、话题跟进

---

## ⏸️ P3 — 后期增强（暂不实现，留文档）

以下功能评估成本/复杂度过高，推迟到验证 P0-P2 用户价值后再议：

| 功能 | 暂缓原因 |
|------|---------|
| 行为记忆（敲击日志 + 规律分析） | 隐式推断误判成本高，需专项 AB 测试 |
| 言行冲突检测 | 滑动窗口算法复杂，V1 用户量不足以验证 |
| 语义检索（向量/分词） | 基础召回够用，过早引入增加维护成本 |
| 话题未结束主动跟进 | "话题有效"定义难统一，误触发率高 |
| 每周行为规律分析 | 需稳定用户行为数据，冷启动无意义 |

## 实现顺序总结

```
P0-1 Schema 迁移方案（文档先行） ✅
  ↓
P0-2 角色记忆隔离（代码改动最小，风险最低） ✅
  ↓
P0-3 时间语义统一（改时间戳存储方式） ✅
  ↓
P0-4 成本上限保护（加 token 计数 + 日限） ✅
  ↓
P1-3 统一触发判定器（重构现有触发逻辑，不改功能） ✅
  ↓
P1-1 结构化档案自动提取（新增摘要后置步骤） ✅
  ↓
P1-2 角色关系状态（新增表 + 注入） ✅
  ↓
P2-2 全局静默开关（一个设置项） ✅
  ↓
P2-3 基础主动发话（2种触发） ✅
  ↓
P2-1 记忆管理界面（最后做，依赖前面数据结构稳定） ✅
```

## 量化验收指标（P2 完成后开始采集）

| 指标 | 目标 | 采集方式 |
|------|------|---------|
| 摘要触发成功率 | ≥ 95% | summary_history 成功/失败比 |
| 主动发话接受率 | ≥ 60% | 触发后用户继续对话 vs 关闭 |
| 主动发话误触发率 | ≤ 10% | 用户手动关闭/静默的比例 |
| 用户档案自动填充准确率 | ≥ 80% | 手动抽查 20 条提取结果 |
| 每会话额外 token 消耗 | ≤ 2000 tokens | 摘要 + 提取调用的 input 计数 |
